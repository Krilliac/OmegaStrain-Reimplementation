## 1. Identity

`.64` is the extension of the PS2 boot executable named per Sony's disc-ID
naming convention (`SCUS-97264` â†’ `SCUS_972.64`). Evidence tracked in this
repository shows it is an ELF32 little-endian MIPS binary â€” the PS2 game's
main executable image, i.e. a **system/boot file**, not a game data asset.
This dossier is therefore a **stub classification**, not a full asset
dossier: no decoder work is warranted because the file is out of scope for
the asset-format fingerprinting pipeline.

## 5. Hypotheses

- **H1**: `SCUS_972.64` is the sole PS2 boot ELF referenced by `SYSTEM.CNF`'s
  `BOOT2=` line. *Confirmation/refutation*: a privacy-safe grep of the
  tracked `SYSTEM.CNF` size/entry already in `disc-summary.json` cannot
  confirm this by itself; confirming would require an aggregate-only
  parse of `SYSTEM.CNF`'s `BOOT2=` field name and target filename token
  (already a generic container name class, not private payload) recorded
  into a tracked manifest, without exposing raw byte offsets tied to the
  private disc image.
- **H2**: No other `.64`-suffixed file will ever appear on the disc or in
  any HOG archive in a hypothetical future re-extraction, because the
  extension encodes a disc-ID-derived boot-executable name rather than a
  general asset-type suffix. *Confirmation/refutation*: re-running the
  existing `analysis/formats/asset-fingerprints.json` / HOG-validation
  scan tooling against the owner corpus and checking that recursive-in-HOG
  and top-level-HOG `.64` counts remain 0 would support H2; a nonzero count
  in either would refute it.

## 7. Decoder/tooling status

**Classification: `system_file_out_of_scope`**

Justification: the tracked evidence (ledger `E-0002`, `E-0004`, and the
disc-manifest entry) mechanically establishes `SCUS_972.64` as a single,
uniquely-named ELF32 MIPS executable â€” the PS2 boot binary â€” not a class of
game data asset. It has zero occurrences inside any HOG archive (recursive
or top-level), it is not referenced by name in any published asset-format
grammar doc, and no entry in `tools/fingerprint_assets.py`'s
`FORMAT_HANDLERS`, no header in `native/include/omega/retail/*.h` or
`native/include/omega/asset/*.h`, and no `CMakeLists.txt` target treats
`.64` as a decodable/registered asset format. The project's existing ELF
inspection tooling (`tools/inspect_ps2_elf.py`) already covers this file at
the structural level the ledger cites (program headers, sections, entry
point) as part of startup/loader research, not as part of the asset
fingerprinting pipeline â€” so no gap exists to fill for asset-dossier
purposes, and building an asset decoder for `.64` would be out of scope and
would risk inventing semantics for a system file the clean-room rules
explicitly ask to leave alone.
