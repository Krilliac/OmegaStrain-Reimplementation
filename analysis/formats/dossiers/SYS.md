# .SYS — PS2 platform system file (out of decoder scope)

**One-line identity:** the `.sys` suffix appears only as a whole-disc file (never inside a HOG archive), at a fixed size that is identical across both observed instances — consistent with a PS2 platform/BIOS-reserved system file rather than a project game asset. No tracked evidence assigns it a game-content role.

## 1. Occurrence evidence

| Scope | Count | Tracked source |
|---|---|---|
| Recursive, inside any HOG archive | 0 | `analysis/formats/asset-fingerprints.json` |
| Top-level HOG member suffix | 0 | `analysis/formats/hog-validation.json` |
| Whole-disc (extracted-disc file histogram) | 2 | `analysis/manifests/disc-summary.json` (`"extensions": {".sys": 2, ...}`) |

`analysis/manifests/disc-summary.json` reports `file_count: 448` and `total_bytes: 3,455,648,408` for the whole-disc corpus; `.sys` is 2 of the 448 files and is one of several suffixes (alongside `.cnf`, `.icn`, `.ico`, `.irx`, `.sfo`-style `.ini`, etc.) that appear only at whole-disc scope and never recur inside `.hog` containers. This whole-disc-only distribution pattern is itself an aggregate fact from the same file, not an inference about any individual entry.

No `analysis/formats/*.md` grammar document (`HOG.md`, `TDX.md`, `COL.md`, `VUM.md`, `POP.md`, `LPD.md`, `PAR.md`, `VAG.md`, `SKAS.md`, `SO.md`, `VPK.md`, `ASSET-RECON.md`, `FRONTEND-TOPOLOGY.md`) defines a `.sys` grammar. `FRONTEND-TOPOLOGY.md`, `PAR.md`, `POP.md`, and `VAG.md` contain incidental text matches on the substring "sys" (e.g. "system") but none establish a `.sys` file format.

## 4. Hypotheses

| # | Hypothesis | Confirming observation | Refuting observation |
|---|---|---|---|
| H1 | The `.sys` suffix denotes a PS2 platform/BIOS-reserved file (e.g. a memory-card save-icon descriptor, a boot/IOP-related system file) rather than a game-content asset, given: whole-disc-only distribution (never HOG-packed), identical size across both instances, and co-occurrence with other platform-suffix files (`.cnf`, `.irx`) in the same histogram. | A privacy-safe structural probe of the tracked corpus's two `.sys` entries' leading bytes/magic against the published PS2 platform system-file conventions already present in some tracked doc, reported only as an aggregate magic/structure match — no per-file bytes disclosed. | If a future tracked grammar doc or ledger entry established a `.sys` game-content grammar (fields, roles) citable from tracked project source, this hypothesis would be refuted. |
| H2 | The two whole-disc `.sys` instances are two independent copies of the same fixed-format platform file placed in two different disc locations (rather than two different `.sys` formats sharing a coincidental extension). | Aggregate confirmation that both instances share identical size (A2) is consistent with this; a further aggregate check of identical/near-identical byte-for-byte content (reported only as "match: yes/no", no bytes) would strengthen it. | Any tracked aggregate showing the two instances have different fixed header magic or structurally incompatible sizes would refute it. |
| H3 | The `.sys` files are unrelated to the tracked `omega::compat` PS2 memory-card container module (C4/C5) — i.e., they are not memory-card *container* images themselves (8 MiB card images), but rather small platform files that would, on real hardware, live *inside* such a card or in the BIOS/boot area. | Their recorded whole-disc size (aggregate, per A2) is far smaller than the fixed 8 MiB card-image size that `ps2_memory_card_image.h` documents for a card container, which is consistent with "file inside a card format," not "card container itself." | A tracked structural probe showing either instance parses as a valid 8 MiB-scale card superblock would refute it (none does, per A2's small identical size). |

## 5. Missing observations

- **No tracked grammar document for `.sys`.** None of `analysis/formats/*.md` defines header magic, field layout, or semantics for this suffix. Collection: author a privacy-safe structural probe (extending `tools/fingerprint_assets.py`-style aggregate scanning, not a new interpretive decoder) that reports only aggregate facts about the tracked corpus's `.sys` instances (leading-byte magic class, fixed/variable size class, ASCII-text-region presence/absence) with no per-file bytes or paths — mirroring how `asset-fingerprints.json` reports other suffixes today.
- **No ledger entry (E-####) addresses `.sys` directly.** Collection: if the structural probe above confirms a stable, generic (platform-standard) envelope shape, record a new ledger entry describing only the aggregate envelope facts (magic class, fixed size, whole-disc-only distribution), explicitly scoped as "no Omega Strain payload semantics claimed," matching the disclaimer pattern already used in `E-0085`.
- **No native decoder/descriptor claims `.sys`.** Because the existing `omega::compat` memory-card module explicitly treats card-resident payload bytes as opaque (C5), there is no tracked codepath that would even opaquely wrap a `.sys` file today. Collection: none required unless a future workstream decides `.sys` decoding is in-scope for a memory-card-import/export feature; if so, that decision and its evidence should be ledgered before any decoder is written.
- **No confirmation of the two instances' relationship to each other.** Only "identical size, different hash" is known (A2). Collection: an aggregate-only diff report ("N bytes differ out of size S," no offsets/bytes) would establish whether they are near-identical templates or substantially different files, without disclosing content.

## 6. Decoder/tooling status

**Classification: `system_file_out_of_scope`**

- No structural handler exists in `tools/fingerprint_assets.py` (`FORMAT_HANDLERS` omits `.sys`; C3) — so `.sys` receives generic aggregate accounting only (size/count), the same treatment as any unhandled suffix.
- No native decoder or descriptor exists for `.sys` under `native/include/omega/retail/*.h` or `native/include/omega/asset/*.h`, and none is registered as a `.sys`-specific target in `CMakeLists.txt` (C4).
- The one native module that is topically adjacent — `omega::compat` PS2 memory-card container/filesystem/export (`ps2_memory_card_image.h`/`.cpp`, `ps2_memory_card_filesystem.h`/`.cpp`, `ps2_memory_card_export.h`/`.cpp`) — is registered and tested (`omega_ps2_memory_card_image_tests`, `omega_ps2_memory_card_filesystem_tests`, `omega_ps2_memory_card_export_tests` in `CMakeLists.txt`; ledger `E-0085`, confirmed). But it targets the **8 MiB card container envelope**, not the small whole-disc `.sys` files, and it explicitly returns any card-resident file payload as opaque, uninterpreted bytes (C5). It therefore does not constitute a decoder, passive descriptor, or even a structural envelope reader *for `.sys` content* — it is out of scope for this family by its own documented disclaimer.
- Adversarial/resource-boundary test gap: not applicable — there is no `.sys`-targeting code path to test. If a future structural probe or decoder is built for `.sys`, it will need its own adversarial/fuzz and resource-boundary tests (oversized/undersized inputs, truncated headers, allocation limits) following the pattern already used in `ps2_memory_card_*_tests.cpp` and the PAR/VPK decoder test suites cited in `E-0092`/`E-0094`.

## 7. Codex work order (ranked, privacy-safe)

1. **Build a `.sys` structural probe (aggregate-only), not a decoder.** Extend `tools/fingerprint_assets.py` with a `.sys` aggregate handler that reports, across the owner corpus, only: count, size-class histogram, leading-byte/magic-class histogram (e.g. "N of M start with pattern X"), and presence/absence of embedded ASCII text runs — no per-file rows, no path/hash output beyond what `asset-fingerprints.json` already emits in aggregate form for other suffixes. This directly tests H1–H3 without violating the privacy/aggregate constraints.
2. **Run the probe against the tracked owner corpus and record results in `analysis/formats/asset-fingerprints.json`** alongside the existing per-format aggregates, so `.sys` gets the same aggregate visibility as the currently-handled suffixes (`.tdx`, `.col`, `.vum`, etc.) even though it stays undecoded.
3. **If (and only if) the probe shows a stable, generic envelope shape** (fixed size class, consistent magic class), draft a ledger entry (`E-00NN`) documenting the aggregate envelope facts only, following the disclaimer pattern in `E-0085`/`E-0092`/`E-0094` ("no Omega Strain payload semantics claimed; no private/owner input accessed"). Do not draft a decoder or IR header at this step.
4. **Do not attempt to identify the two whole-disc `.sys` instances' specific platform role (boot config vs. save icon vs. IOP-related) beyond what the aggregate probe can support.** Any such identification must remain a labeled Hypothesis (per Section 5) until a privacy-safe aggregate structural match is recorded in a tracked source — this avoids the "plausible invented decoder or semantic" regression the project's clean-room rules flag explicitly.
5. **Leave `FORMAT_HANDLERS` and the native `omega::compat`/`omega::retail` decoder set unchanged for `.sys`** unless step 3's ledger entry is accepted and a follow-up work order explicitly authorizes writing a structural-envelope-only reader (mirroring the `Ps2MemoryCard*` opaque-payload pattern) — never a payload-interpreting decoder, since no tracked evidence assigns `.sys` payload semantics.
